# KIR Data Pipeline and Dashboard Design

**Date:** 2026-04-29

## Purpose

This project processes KIR Excel datasets by joining KIR source data with poteri data for two routes, producing a trusted analytical dataset while preserving auditability. The primary risk to avoid is accidental data loss during merge, transformation, cleaning, or outlier handling.

## Core Principle

Data must not disappear silently.

The pipeline must preserve a raw merge artifact for audit and comparison. The main analytical output may add quality flags and diagnostics, but it must not remove analytically debatable rows such as zero values, missing poteri matches, or statistical outliers.

## Outputs

Each pipeline run must produce these files per route.

When a single command processes multiple routes, each route must write into its own run directory, for example:

- `data/run_12_route_1/merged_raw.xlsx`
- `data/run_13_route_2/merged_raw.xlsx`

The file names inside a route run directory may be the same because the directory carries the route/run identity.

1. `merged_raw.xlsx`
   - Full raw merge result.
   - Created before any cleaning or analytical transformation.
   - Used for audit and comparison.
   - Must not be overwritten by cleaning steps.

2. `final_clean_data.xlsx`
   - Main working analytical file.
   - Contains the merged data plus quality/diagnostic flags.
   - Keeps rows without poteri matches.
   - Keeps zero values.
   - Keeps outliers.
   - Does not depend on a configured `target_col`.

3. `excluded_rows.xlsx`
   - Contains only rows that cannot be represented even as flagged analytical rows.
   - Must include `exclude_reason`.
   - Expected to be small or empty.
   - Missing merge keys are not excluded by default; they stay in `final_clean_data.xlsx` with `has_missing_key=true`.

4. `merge_diagnostics.md` or `merge_diagnostics.xlsx`
   - Summarizes merge counts, matches, missing matches, duplicate keys, excluded rows, and final-vs-raw row counts.

5. `outliers_report.xlsx`
   - Optional analytical report generated by the dashboard or by an explicit manual analysis mode.
   - The default pipeline must not require outlier columns or a target metric.
   - Outliers are marked/reported, not removed by default.

## Target Metric / target_col Decision

`target_col` must not be required for the pipeline.

The pipeline must not use a configured target metric to decide which rows to keep or remove. It should preserve all KIR/metric columns from the KIR file and attach poteri columns.

Metric selection belongs in the dashboard, not in the pipeline.

The dashboard should discover available numeric columns from `final_clean_data.xlsx`, prioritize columns whose names start with `КИР-`, and let the user select which metric to analyze.

A future optional dashboard default may exist, for example:

```yaml
dashboard:
  default_metric: null
```

But this must not affect pipeline output.

## Final Clean Data Rules

`final_clean_data.xlsx` is the main analytical output, but "clean" means analytically prepared and flagged, not aggressively filtered.

Rows must remain in `final_clean_data.xlsx` when:
- poteri data was not matched;
- selected or potential KIR metric is zero;
- writeoffs/revenue/stock are zero;
- KIR units are zero;
- metric values are statistical outliers;
- values are unusual but technically valid.

Rows with missing merge keys must stay in `final_clean_data.xlsx` whenever they can be represented as rows. They must be flagged:

- `has_missing_key=true`
- `quality_status="warning"` or equivalent
- `quality_reason` includes the missing key names

Rows may be excluded only when they are technically impossible to represent after loading/normalization, for example:

- unreadable/corrupt source row;
- row cannot be represented after normalization due to structural corruption;
- row would break Excel output serialization.

A missing metric value is not grounds for exclusion at pipeline level, because metric selection happens in the dashboard.

A zero metric value is valid.

## Quality Flags

`final_clean_data.xlsx` should include quality and diagnostic columns such as:

- `has_poteri_match`
- `has_duplicate_kir_key`
- `has_duplicate_poteri_key`
- `has_missing_key`
- `has_zero_writeoffs`
- `has_zero_revenue`
- `has_zero_stock`
- `has_zero_kir_units`
- `quality_status`
- `quality_reason`

Outlier-related fields can be included when outlier analysis is run:

- `is_outlier`
- `outlier_columns`
- `outlier_reason`

## Cleaning and Outlier Policy

The previous cleanup behavior is too risky because it physically removes rows from the main output.

New default policy:
- no deletion of zero values;
- no deletion of rows without poteri match;
- no deletion of rows with missing merge keys when they can be represented;
- no deletion of outliers;
- mark questionable rows with flags;
- write excluded rows only for structurally unrepresentable rows.

Outlier detection must be mark-first and metric-driven.

The default pipeline should not calculate outliers because the pipeline no longer has a mandatory `target_col`.

Outliers can be calculated in two supported places:

1. Dashboard analysis
   - The user selects a numeric metric in the Streamlit UI.
   - The dashboard calculates outlier flags for the filtered view.
   - The dashboard can export `outliers_report.xlsx` for the selected metric and filters.

2. Explicit manual/batch analysis mode
   - A config key such as `cleaning.outlier_columns` explicitly lists columns.
   - If the list is empty or absent, no pipeline outlier report is generated.

Example:

```yaml
cleaning:
  outlier_columns: []
  drop_outliers: false
```

Physical deletion remains disabled by default.

If physical outlier removal is ever needed, it must be explicit:

```yaml
cleaning:
  drop_outliers: false
```

Default must remain `false`.

## Merge Diagnostics

Every run should report:

- KIR input row count;
- poteri input row count;
- raw merge row count;
- final clean row count;
- excluded row count;
- count of rows with missing merge keys;
- count of rows with poteri match;
- count of rows without poteri match;
- duplicate KIR keys;
- duplicate poteri keys;
- duplicate key group sizes for KIR;
- duplicate key group sizes for poteri;
- final-vs-raw row difference;
- route name;
- merge key used.

The diagnostics must make it easy to answer: "Did anything disappear, duplicate, or fail to match?"

Audit invariants:

- every KIR source row must get a stable `source_row_id` before merge;
- `merged_raw.xlsx` must preserve one row per KIR source row unless duplicate poteri keys intentionally create row multiplication;
- if duplicate poteri keys create row multiplication, diagnostics must report the affected merge keys and multiplication counts;
- `final_clean_data.xlsx` plus `excluded_rows.xlsx` must account for all rows from `merged_raw.xlsx`;
- missing poteri matches must be represented by a merge indicator such as `has_poteri_match=false`;
- missing merge keys must be represented by `has_missing_key=true`, not silently dropped;
- diagnostics must include `raw_row_count`, `final_row_count`, `excluded_row_count`, and `final_row_count + excluded_row_count == raw_row_count`.

## Dashboard Responsibility

The dashboard is the analytical layer. It should open `final_clean_data.xlsx` and optionally reference `merged_raw.xlsx` for audit comparison.

The dashboard must let the user choose which metric/column to analyze. The pipeline should not preselect or hardcode this.

## Dashboard MVP

The MVP dashboard should include:

1. Run/file selection
   - choose latest run;
   - choose route;
   - display paths to `final_clean_data.xlsx` and `merged_raw.xlsx`.

2. Metric selection
   - dropdown with numeric columns;
   - KIR columns shown first;
   - no dependency on `target_col`.

3. Filters
   - week;
   - `ТС`;
   - category for route_1;
   - factory/store (`Завод`);
   - `has_poteri_match`;
   - quality flags.

4. Data quality overview
   - total rows;
   - raw-vs-final row count;
   - matched poteri rows;
   - unmatched poteri rows;
   - duplicate keys;
   - missing keys;
   - zero count for selected metric;
   - NaN count for selected metric.

5. Metric analysis
   - histogram;
   - boxplot;
   - count;
   - min;
   - max;
   - mean;
   - median;
   - p25;
   - p85;
   - zero share;
   - missing share.

6. Group comparison
   - group by `ТС`;
   - group by `Категория` for route_1;
   - group by `Завод`;
   - show count, mean, median, p85, sum where applicable.

7. Poteri relationship analysis
   - selected metric vs `Списания`;
   - selected metric vs `Выручка`;
   - selected metric vs `Свободный ТЗ`;
   - scatter plots and correlations.

8. Problem rows
   - rows without poteri match;
   - duplicate-key rows;
   - missing-key rows;
   - outlier-flagged rows;
   - filtered table export to Excel/CSV.

The dashboard must read `final_clean_data.xlsx` for missing-key rows because these rows remain in the main analytical file with flags. It may also read `excluded_rows.xlsx`, but only to show structurally unrepresentable rows that could not be retained in final data.

## Dashboard Technology Decision

Use Streamlit for the analytical dashboard.

Rationale:
- project is already Python/pandas/Excel based;
- Streamlit is faster and simpler than Tkinter for filters, charts, tables, and local analytical workflows;
- Dash is more flexible but unnecessarily heavy for this project right now;
- Tkinter is insufficient for the intended analytical UX.

Expected run command:

```powershell
streamlit run dashboard_streamlit.py
```

The pipeline and dashboard should be separate:

```powershell
python main.py route_1
python main.py route_2
streamlit run dashboard_streamlit.py
```

This is the target architecture, not the current repository state.

Migration steps:

- keep `main_final_v3.py` working during migration;
- make `main.py` the supported wrapper around the final pipeline entrypoint;
- remove or archive broken legacy entrypoints (`main_automated.py`, `main_final.py`, `main_final_v2.py`, `main_new.py`);
- replace the temporary Tkinter dashboard with `dashboard_streamlit.py`;
- remove the hard dependency from the pipeline to `scripts.dashboard_tkinter`.

## Non-Goals For First Fix

- Do not build complex user authentication.
- Do not deploy the dashboard to a server.
- Do not physically remove outliers from the main output.
- Do not use `target_col` as a pipeline filter.
- Do not optimize away rows for file size or speed before auditability is guaranteed.

## Implementation Implications

The existing fix plan should be updated to reflect this design:

- replace `merged_step1.xlsx` with `merged_raw.xlsx` as immutable audit output;
- stop writing pre-clean results back into the raw merge file;
- remove mandatory `target_col` from merge and cleaning code;
- add quality flags instead of row deletion;
- make outlier handling mark-first;
- add Streamlit dashboard as a separate app;
- add tests proving raw row preservation and no zero-value deletion.
